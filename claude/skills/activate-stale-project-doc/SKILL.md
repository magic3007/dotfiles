# 激活停滞的项目文档

## Problem

项目文档沦为"空壳"：状态是 active，但周记录全空、里程碑是模板占位符、工作日志只有 onboarding 第一天。用户每天都在工作，但由于文档没有运转起来，无法直观看到累积的进展，产生"出不了成果"的焦虑。

## Context / Trigger Conditions

出现以下任意一条即可触发：
- 项目已进行 N 周，但周记录中空白行数 > 50%
- 里程碑全为 `{{占位符}}` 或未勾选的任务，没有已完成标记
- 工作日志只有项目开始第一天的一条记录
- 用户说"做了很多但感觉没产出"或"不知道怎么出成果"

## Solution

### Step 1: 诊断空壳范围

快速扫描项目文档的三个核心区域：
1. **Dashboard**：`已进行 X / Y 周` 是否与实际时间匹配？
2. **周记录**：空白行占比多少？
3. **里程碑**：是否有已完成的阶段标记？是否有未来阶段的规划？

### Step 2: 从已有记录反推补全

不要从零开始写，而是从用户已有的工作痕迹中提取：
- **周日记**（如 life-os 的 `diary/YYYY-MM-DD-weekly.md`）：提取每周的关键产出
- **任务管理工具**（如滴答清单、Notion）：按项目过滤已完成任务
- **代码提交 / 实验记录**：按时间线整理技术动作
- **飞书 / Slack 汇报**：提取已有的阶段性总结

补全优先级：
1. 先补 **周记录**（宏观时间线）
2. 再补 **工作日志**（微观任务列表）
3. 最后补 **里程碑**（基于已做+待做的事定义阶段）

### Step 3: 定义阶段性里程碑

基于项目剩余时间和目标，将剩余工作拆分为 3-5 个阶段。典型 research intern 的五阶段模板：

| 阶段 | 目标 | 时间 |
|:-----|:-----|:-----|
| M1 熟悉与环境 | 跑通第一个实验，完成 onboarding | W1-W2 |
| M2 数据/框架积累 | 建立评测流程，积累实验数据 | W3-W5 |
| M3 机制深度分析 | 定位瓶颈，产出分析报告 | W6-W7 |
| M4 方案验证 | 实现改进，跑实验，确认指标提升 | W8-W9 |
| M5 成果产出 | 论文/专利/技术报告初稿与交付 | W10+ |

关键原则：
- 每个阶段必须有 **可验证的产出**（报告/实验结果/PR）
- 当前阶段用 🔄 标记，已完成的用 ✅，未来的用 ⏳
- 阶段任务要细化到周级别，不要只写"分析"而要写"产出 X 页分析报告"

### Step 4: 建立同步闭环

确保日常记录能自动流入项目文档：
1. **每日**：完成任务后即时在任务管理工具中勾选
2. **每晚**：运行 `/daily-review`（或等价的同步命令），自动同步到项目工作日志
3. **每周日**：10 分钟三件套——更新周记录、检查里程碑进度、创建下周关键任务
4. **每月末**：对照年度目标调整里程碑优先级

## Verification

激活成功的标志：
- [ ] 项目文档的"已进行周数"与"有记录周数"的差距缩小到 ≤ 1 周
- [ ] 里程碑中至少有一个阶段被标记为 ✅，当前阶段被标记为 🔄
- [ ] 用户能在一分钟内说出"本周处于哪个阶段、下周要推进哪个里程碑"

## Example

**激活前**（ResearchAgent.md）：
- 已进行：0 / 13 周（实际已到第 6 周）
- 周记录：13 行全空
- 里程碑：`{{里程碑1名称}}` 占位符
- 工作日志：只有 2026-03-23 "看完文档"

**激活后**：
- 已进行：6 / 13 周，当前阶段"机制深度分析"
- 周记录：W1-W5 已补全关键产出（PaperBench 评测、trace 回捞、FCG 博客）
- 里程碑：M1 ✅ / M2 ✅ / M3 🔄 / M4 ⏳ / M5 ⏳
- 工作日志：4.19-4.29 每天都有任务记录

## Notes

- **不要从零写文档**。用户的挫败感往往来自"要写太多"，解决方法是先回填已有记录，降低启动门槛。
- **里程碑要跟 mentor 对齐**。自己定义的路线图必须和 mentor 的预期一致，否则再完美的规划也是自嗨。
- **空白行不可怕，可怕的是不知道下一步填什么**。定义了清晰的 M3-M5 后，每天的任务就有了归属。
